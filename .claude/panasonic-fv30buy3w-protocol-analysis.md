# Panasonic FV-30BUY3W 協議逆向工程 — 最終報告

## 結論

逆向工程完成。22 個指令封包 + 8 個穩態回報封包全部解碼並交叉驗證。
ESP32 可用查表法重放封包控制主機，不需要計算 checksum。

---

## 通訊架構

```
面板 (master) ──── 單線半雙工 ──── 主機 (slave)
   │                                    │
   ├─ 定期發送 polling 封包              ├─ 收到 polling 後回應當前狀態
   ├─ 按鍵時發送指令封包取代 polling      ├─ 收到指令後切換模式
   └─ 面板不在時，data line 為 idle      └─ 主機不會主動發送任何東西
```

面板是 master，主機是 slave：

- 面板是通訊發起者：100% 確認（拔面板後 data line 靜默）
- 封包方向：95% 確認（時序+按鍵行為推論，未用雙通道監聽確認）

## 開機時間線（ele.sr 實測）

```
復電 +0.00s   Data line 從 LOW 拉到 HIGH
復電 +0.44s   面板發出第一個 polling
復電 +2.46s   面板第二個 polling（第一個無回應，重試間隔 ~2 秒）
復電 +3.18s   主機首次回應「待機」
復電 +3.90s   正常交替通訊開始（~720ms 週期）
```

- 面板開機時間：< 0.5 秒
- 主機開機時間：~3 秒
- 無 handshake，面板直接開始 polling
- 指令生效延遲：1 個通訊週期（~700ms）

## 驗證記錄

### 2026-04-07 從原始 .sr 檔重新驗證（analyze_captures.py）

從 PulseView .sr 原始錄製檔（24MHz 取樣率）重新解析並比對 packets.json：

| 錄製檔                       | 匹配率         | 說明                     |
| ---------------------------- | -------------- | ------------------------ |
| 換氣(15m~24h~15m).sr         | 26/28 (92.9%)  | 首尾封包因錄製截斷       |
| 取暖-all-in-one.sr           | 149/154 (96.8%) | 首封包截斷 + 量化雜訊   |
| 乾燥熱-all-in-one.sr         | 185/193 (95.9%) | 首封包截斷 + 量化雜訊   |
| 乾燥涼-all-in-one.sr         | 180/181 (99.4%) | 1 個量化雜訊             |

**結論**：所有未匹配封包均為錄製截斷（首尾）或量化邊界雜訊（長串 `1,1,1,...`），**無真正的未知封包類型**。packets.json 中的所有封包均已由原始錄製資料確認。

### 2026-04-07 H/L 極性修正

發現文件中 waveform 格式標示錯誤：原先標記為 even=HIGH, odd=LOW，實際應為 **even=LOW, odd=HIGH**。封包的第一個動作是從 idle HIGH 拉低 data line，因此 waveform[0] 是 LOW 持續時間。此錯誤導致 ESP32 送出極性反轉的信號，主機完全不回應。修正後通訊正常。

### 時序實測值（從原始錄製提取）

| 項目               | 實測值              | 文件值     |
| ------------------ | ------------------- | ---------- |
| 封包持續時間       | 587~620ms           | ~590/610ms |
| 封包間隔           | 93~135ms            | ~130ms     |
| 最大間隔（開機等待）| 3.5s~8.6s          | ~3.2s      |

### 封包格式說明

- **packets.json**: 交替 `[L_dur, H_dur, L_dur, H_dur, ...]`，完整 waveform（偶數 index = LOW，奇數 index = HIGH）
- **protocol-analysis.md 查表**: 僅列 LOW 持續時間 `[L, L, L, ...]`（即 packets.json 偶數 index）
- packets.json 的偶數 index (0, 2, 4, ...) = LOW 持續時間（主動拉低 data line）
- packets.json 的奇數 index (1, 3, 5, ...) = HIGH 持續時間（釋放由 pull-up 拉高）
- 封包從 LOW 開始（第一個動作是將 idle HIGH 拉低），以 LOW 結束後回到 idle HIGH
- **2026-04-07 修正**: 原先文件誤標為 [H, L, H, L, ...]（odd=LOW, even=HIGH），導致 ESP32 送出極性反轉信號。已全面修正為 even=LOW, odd=HIGH

## 面板行為備註

- 面板會記住每個模式最後的定時設定
- 乾燥鍵是 toggle，但**會記住上次最後使用的是熱風或涼風**（不一定先熱風）
- 面板是 master，主機是 slave
- 指令封包同時包含模式+定時資訊，不是分開的
- 主機回應不含模式資訊，ESP32 需自己追蹤模式
- Checksum 算法未破解，不影響實作（查表法）
